# 开发日志

## 当前阶段进度（2026/3/29-2026/3/30）

### 本阶段目标
这一阶段的核心目标，是把项目从一个空的 Unity 2D 工程，推进到一个可以实际测试核心玩法的原型阶段。

目前重点集中在三件事上：
- 搭建角色的基础控制与挥剑反弹原型
- 建立可扩展的房间式地图数据结构
- 制作一套可以直接用于内容生产的房间编辑器

### 已完成内容

#### 1. 玩家基础控制器
目前已经完成了一套自定义的 2D 玩家控制器，没有把角色运动完全交给 Unity 刚体物理。

当前支持：
- 左右离散移动
- 空中横向控制
- 重力下落
- 最大下落速度限制
- 地面接地检测与贴地修正
- 横向停下时按 `0.1` 单位对齐位置
- 可单独调整角色实际碰撞体大小

这一套控制器的目的，不是做传统平台跳跃，而是为后续“挥剑命中后反弹移动”的核心玩法服务。

#### 2. 输入系统
项目已经切换到 Unity 的 `Input System`。

当前输入方案已经统一支持键盘、鼠标与手柄：
- `WASD` / 左摇杆 / 十字键：移动
- 鼠标左键 / 空格 / 手柄 `A`：主能力
- 鼠标右键 / 手柄 `X`：副能力
- `Tab` / 手柄 `LB`：地图相关输入

输入层已经单独拆出，不和角色移动、挥剑逻辑硬绑定，后续做重绑定或扩展输入时会更容易。

#### 3. 挥剑系统
挥剑逻辑已经从角色控制器中拆出，形成独立的武器组件。

当前已实现：
- 四方向挥剑判定
- 不按上下时按最后左右朝向攻击
- 按上时向上砍
- 按下时向下砍
- 剑的命中检测
- 命中后触发角色反弹
- 攻击判定盒调试可视化

这部分目前的重点是先验证玩法和手感，而不是最终表现。

#### 4. 反弹 / Recoil 原型
项目的核心移动机制已经开始围绕“挥剑命中后的反弹”构建。

当前已实现：
- 命中碰撞体后按攻击反方向反弹
- 水平反弹与竖直反弹分开配置
- 可调整反弹距离、持续时间、曲线、退出速度
- 水平反弹期间可锁定横向输入
- 竖直反弹期间可允许横向控制
- 反弹与普通移动系统分离，方便后续继续打磨

这部分已经是玩法原型的核心之一，但手感仍然处于持续调整阶段。

#### 5. 单向平台
地图中已经加入新的格子类型 `OneWayPlatform`。

当前单向平台具备这些基础规则：
- 显示在单个格子的上半部分
- 玩家可以从下方穿过
- 玩家可以从上方站上去
- 玩家可以用剑砍到单向平台
- 在特定反弹状态下，可短时间穿过单向平台

目前单向平台已经能参与运行时生成与测试，但它的碰撞细节和视觉一致性仍然需要继续打磨。

#### 6. 房间地图数据结构
项目地图已经不是直接手工堆在 Scene 里，而是开始使用数据驱动方式组织。

当前结构为：
- 一个 `MapDefinition` 作为整张地图的数据容器
- 多个 `MapRoomDefinition` 作为独立房间数据
- 每个房间固定尺寸为 `20 x 15`

每个房间当前可保存：
- 房间 ID
- 房间名
- 是否允许在地图中显示
- 每个格子的格子类型

当前支持的格子类型：
- `Empty`
- `Wall`
- `Ground`
- `OneWayPlatform`

#### 7. 运行时地图生成
编辑器中编辑的房间内容，已经能够生成到运行时场景中进行测试，而不是只停留在预览层。

当前生成内容包括：
- 可见格子
- 实际碰撞体
- 单向平台标记

也就是说，编辑器中画出来的地面与墙体，已经可以被玩家直接用于测试移动、碰撞和挥剑。

#### 8. 基于房间的相机与视口方案
目前已经有基础的房间相机控制方案。

当前支持：
- 根据玩家位置识别其处于哪个房间
- 相机锁定到当前房间中心
- 所有房间保持激活，不会因为切换房间而关闭
- 通过视口遮罩方案，仅显示当前房间区域

这样做的原因，是为了兼容“全图机关可能同时生效”的设计需求。

#### 9. 地图编辑器
项目已经有一套自定义地图编辑工作流，而不是完全依赖原生 Tilemap 手工搭建。

当前已完成：
- 地图编辑根对象 `MapAuthoringRoot`
- 房间资源自动创建
- 房间在世界中的放置
- 房间内容的实时编辑
- 编辑结果实时刷新到运行时测试层

同时，重复在同一坐标新建房间时，已经开始朝“优先复用同名房间资源”的方向收口，避免无意义的重复资源堆积。

#### 10. 独立的 Room Editor 窗口
除了 Scene 内编辑外，目前也已经实现了独立的 `Room Editor` 编辑窗口。

当前窗口已支持：
- 编辑当前房间的 `20 x 15` 网格
- 左键绘制、右键擦除
- 使用 `1 / 2 / 3 / 4` 快捷键切换笔刷
- 直接编辑房间名
- 左侧编辑当前房间
- 右侧查看粗略世界总览

右侧世界总览当前支持：
- 按房间在世界中的相对位置进行展示
- 选中区域
- 在空区域创建新房间
- 用当前房间覆盖某区域
- 删除某区域的房间
- 右键拖动查看其他区域

这套工具的目标，是逐步形成适合大型项目持续扩展的地图制作工作流。

### 当前存在的问题
虽然目前已经进入可测试原型阶段，但还有不少部分只是“可用”，还没有达到稳定状态。

当前比较明确的问题有：
- 单向平台的碰撞与表现仍需进一步统一
- 反弹手感仍需要持续打磨
- 世界总览与房间编辑器的交互还需要继续优化
- 房间视口遮罩现在是功能优先，还不是最终表现方案
- 地图系统目前还没有加入正式的房间连接、门、存档与探索 UI

### 当前阶段的意义
到目前为止，项目已经完成了从“空工程”到“可玩原型骨架”的过渡。

现在已经具备：
- 一个可实际测试的角色控制原型
- 一套围绕挥剑反弹构建的核心玩法基础
- 一个可扩展的房间地图数据结构
- 一条从地图编辑到运行时测试的完整工作流

这意味着接下来不再只是验证“能不能做”，而是可以开始持续验证：
- 关卡是否好玩
- 房间结构是否合理
- 反弹机制是否真的能支撑完整地图设计

### 下一步计划
接下来的重点会继续围绕“把原型打磨成稳定的可扩展基础”展开。

下一步优先方向：
- 继续打磨单向平台与反弹相关规则
- 完善房间连接与切换逻辑
- 补充地图探索与显示机制
- 优化编辑器交互体验
- 持续打磨玩家控制和相机表现


## 开发进度更新（2026/3/31）

### 本次重点
今天的开发重点主要放在三个方向上：
- 为玩家加入第二个核心能力“长枪”
- 继续扩展地图格子类型与对应的运行时规则
- 进一步整理编辑器与能力系统的代码结构

### 已完成内容

#### 1. 第二核心能力：长枪
目前已经完成了一版可测试的长枪原型，作为玩家的第二个核心能力接入。

当前已实现：
- 鼠标右键 / 手柄 `X` 触发长枪能力
- 短按：按当前四方向直接投出长枪
- 长按：进入子弹时间，松开发射长枪
- 长枪飞行、插入、牵引、提前取消牵引、破碎、回收这一整套状态链
- 长枪飞行超过 `3` 秒时自动破碎
- 长枪插入表面后，露在外面的长度固定为 `1` 格
- 长枪插住后再次按键，可将玩家向长枪位置牵引
- 牵引过程中再次按键，可提前终止并保留当前方向速度
- 牵引过程中撞到障碍时，长枪破碎并结束牵引

这意味着项目的“第二能力”已经不再只是概念，而是进入了可反复测试和继续调手感的阶段。

#### 2. 长枪与单向平台规则
长枪插入后的交互继续向正式玩法靠拢。

当前补充完成：
- 水平插入的长枪可以作为单向平台使用
- 插住的长枪可以被剑命中
- 被剑命中后，玩家会正常反弹，同时长枪破碎
- 单向平台的有效挥砍方向开始按“平台朝向”判断，而不是统一处理

具体来说：
- 水平单向平台：只有上下砍才有效
- 竖直单向平台：只有左右砍才有效

这样做是为了让普通单向平台和“插在墙上的长枪平台”都能复用同一套规则，而不是后面不断堆特判。

#### 3. 新的地图格子类型
今天继续扩展了地图编辑器可用的格子种类，并将它们接到了运行时逻辑中。

新增格子包括：
- `Death`
- `SavePoint`

对应规则为：
- 玩家碰到死亡格子后会死亡
- 玩家死亡后会从当前唯一激活的存档点复活
- 若当前没有激活存档点，则回到初始出生位置
- 存档格子本身没有实体碰撞，但会作为触发区域激活存档

同时还补充了以下联动规则：
- 飞行中的长枪碰到死亡格子会破碎
- 剑砍死亡格子不会反弹
- 玩家死亡时，如果场上还有长枪，则长枪会一并破碎回收

#### 4. 编辑器支持与快捷键
房间编辑器也随着新格子的加入同步扩展。

当前已补充：
- `Death` 与 `SavePoint` 的可视化绘制
- 快捷键：
  - `5` = `Death`
  - `6` = `SavePoint`
- 房间名称可直接在 `Room Editor` 中修改
- 世界总览中的区域选择、创建、替换、删除流程继续完善

目前房间编辑器已经逐渐从“能用的原型工具”向“适合长期扩展的内容生产工具”靠近。

#### 5. 能力解锁入口
考虑到这两个核心能力都需要随着关卡推进逐步获得，今天也把“能力解锁”从临时逻辑整理成了正式入口。

当前已完成：
- 给剑与长枪都加入了可锁定 / 可解锁接口
- 新增统一入口 `PlayerAbilityLoadout`
- 玩家现在可以在开局时没有剑或没有长枪
- 后续可以通过关卡事件、触发器或剧情逻辑单独开启对应能力

这一步的意义在于：后面做能力获取、能力门、能力成长时，不需要再直接操作具体脚本内部状态。

#### 6. 代码重构
今天也做了一轮结构整理，重点是把容易继续膨胀的部分拆出职责边界。

本次重构包括：
- 为玩家能力引入统一解锁接口 `IPlayerUnlockableAbility`
- 让 `PlayerSword` 与 `PlayerSpear` 共享这套能力启用/禁用协议
- 将 `Room Editor` 中的主题色与格子配色抽到独立的编辑器主题工具
- 将世界总览的边界计算、房间矩形换算、区域命中逻辑抽到独立工具类

这类整理并不会立刻带来新玩法，但会直接降低后续继续加规则时的耦合度。

### 当前存在的问题
虽然今天补上了不少玩法链路，但目前仍然还有明显需要继续打磨的部分：
- 长枪平台与单向平台的碰撞细节仍然需要继续修整
- 长枪牵引的速度曲线与整体手感还处在原型阶段
- 地图编辑器虽然已经更顺手，但世界总览仍有继续优化空间
- `PlayerController2D` 仍然是当前项目里最大的核心文件，后面还需要专门再做一次拆分

### 本阶段的意义
今天最大的推进，不是单独多了一个“长枪功能”，而是项目开始形成“双核心能力 + 地图格子规则 + 编辑器生产工具”三者联动的结构。

这意味着接下来可以开始更认真地验证：
- 两个核心能力是否能够真正支撑关卡设计
- 不同格子与能力之间的组合是否足够丰富
- 当前编辑器工作流是否已经足以支撑后续持续做内容

### 下一步计划
接下来优先考虑继续推进这些方向：
- 继续打磨长枪的飞行、插入、牵引与破碎手感
- 继续完善单向平台与长枪平台的边界交互
- 补充能力获取触发器与能力门逻辑
- 进一步拆分 `PlayerController2D`，降低核心控制器的复杂度
- 持续优化 `Room Editor` 和世界总览的内容制作体验
